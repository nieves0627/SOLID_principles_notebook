1. Single-Responsibility Principle (SRP)#

The single-responsibility principle states that:

A class should have only one reason to change.

This means that a class should have only oneÂ responsibility, as expressed through its methods. If a class takes care of more than one task, then you should separate those tasks into separate classes.

This principle is closely related to the concept ofÂ separation of concerns, which suggests that you should split your programs into different sections. Each section must address a separate concern.

To illustrate the single-responsibility principle and how it can help you improve your object-oriented design, say that you have the followingÂ FileManagerÂ class:

        self.path.write_text(data, encoding)

  • It opens the file specified by the path in text mode.

  • It reads the entire contents of the file as a string.

  1. What does path.with_suffix() method do?

Now you have two smaller classes, each having only a single responsibility.Â FileManagerÂ takes care of managing a file, whileÂ ZipFileManagerÂ handles theÂ compressionÂ andÂ decompressionÂ of a file using the ZIP format. ðŸ’¡ The concept of responsibility in this context may be pretty subjective. Having a single responsibility doesnâ€™t necessarily mean having a single method. Responsibility isnâ€™t directly tied to the number of methods but **to the core task that your class is responsible for**, **depending on your idea of what the class represents in your code**. 2. Open-Closed Principle (OCP)#

Software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification.

To understand what the open-closed principle is all about, consider the followingÂ ShapeÂ class:

3. Liskov Substitution Principle (LSP)#

Subtypes must be substitutable for their base types.

For example, if you have a piece of code that works with aÂ ShapeÂ class, then you should be able to substitute that class with any of its subclasses, such asÂ CircleÂ orÂ Rectangle, without breaking the code.

In practice, this principle is about making your subclasses behave like their base classes without breaking anyoneâ€™s expectations when they call the same methods. To continue with shape-related examples, say you have aÂ RectangleÂ class like the following:

# shapes_lsp.py

    def __init__(self, width, height):
        self.width = width
        self.height = height

    def calculate_area(self):
        return self.width * self.height

InÂ Rectangle, youâ€™ve provided theÂ .calculate_area()Â method, which operates with theÂ .widthÂ andÂ .heightÂ instance attributes.

Because a square is a special case of a rectangle with equal sides, you think of deriving aÂ SquareÂ class fromÂ RectangleÂ in order to reuse the code. Then, you override theÂ setterÂ method for theÂ .widthÂ andÂ .heightÂ attributes so that when one side changes, the other side also changes:

4. Interface Segregation Principle (ISP)#

Clients should not be forced to depend upon methods that they do not use. Interfaces belong to clients, not to hierarchies.

In this case,Â clientsÂ are classes and subclasses, andÂ interfacesÂ consist of methods and attributes. In other words, if a class doesnâ€™t use particular methods or attributes, then those methods and attributes should be segregated into more specific classes.

Consider the following example of classÂ hierarchyÂ to model printing machines:

# printers_isp.py

    @abstractmethod
    def print(self, document):
        pass

    @abstractmethod
    def fax(self, document):
        pass

    @abstractmethod
    def scan(self, document):
        pass

class OldPrinter(Printer):
    def print(self, document):
        print(f"Printing {document} in black and white...")

    def fax(self, document):
        raise NotImplementedError("Fax functionality not supported")

    def scan(self, document):
        raise NotImplementedError("Scan functionality not supported")

    def print(self, document):
        print(f"Printing {document} in color...")

    def fax(self, document):
        print(f"Faxing {document}...")

    def scan(self, document):

    @abstractmethod
    def print(self, document):

    def fax(self, document):
        print(f"Faxing {document}...")

    def scan(self, document):
        print(f"Scanning {document}...")

NowÂ Printer,Â Fax, andÂ ScannerÂ are base classes that provide specific interfaces with a single responsibility each. To createÂ OldPrinter, you only inherit theÂ PrinterÂ interface. This way, the class wonâ€™t have unused methods. To create theÂ ModernPrinterÂ class, you need to inherit from all the interfaces. In short, youâ€™ve segregated theÂ PrinterÂ interface.

This class design allows you to create different machines with different sets of functionalities, making your design more flexible and extensible.

5. Dependency Inversion Principle (DIP)#

Abstractions should not depend upon details. Details should depend upon abstractions.

That sounds pretty complex. Hereâ€™s an example that will help to clarify it. Say youâ€™re building an application and have aÂ FrontEndÂ class to display data to the users in a friendly way. The app currently gets its data from a database, so you end up with the following code:

You may think of adding a new method toÂ BackEndÂ to retrieve the data from the REST API. However, that will also require you to modifyÂ FrontEnd, which should be closed to modification, according to theÂ open-closed principle.
